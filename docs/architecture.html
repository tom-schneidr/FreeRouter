FreeRouter Architecture Roadmap

FreeRouter is moving from a pragmatic Python-served HTML utility to a desktop-first product architecture: FastAPI for the gateway core, React and TypeScript for the control plane, and Tauri for a packaged desktop shell.

React, Tauri, sidecar packaging, and app-data runtime paths are now in place.
Gateway Core FastAPI keeps the OpenAI-compatible API, routing engine, provider adapters, quota state, and model catalog.
Control Plane UI React, TypeScript, Vite, TanStack Query, and typed API clients replace Python string HTML pages.
Desktop Shell Tauri owns windows, tray actions, dynamic localhost port selection, app data paths, and the bundled backend sidecar.

Workspace Shape

apps/ui/          React + TypeScript + Vite frontend
apps/desktop/     Tauri desktop shell
app/               FastAPI backend, sidecar entrypoint, runtime path helpers
docs/              User-facing HTML documentation
scripts/           Development and packaging helpers

Build And Run

npm run dev            Start the backend and React control plane together
npm run dev:web        Start only the React control plane during UI development
npm run dev:desktop    Start the Tauri shell against the backend-served desktop app
npm run build:web      Compile the React control plane into apps/ui/dist
npm run check:desktop  Compile-check the Tauri shell
npm run build:sidecar  Build dist-sidecar/freerouterd-*.exe with PyInstaller
npm run build:desktop  Build the sidecar, React UI, and Tauri desktop bundle

Packaged Desktop Runtime

The installed desktop executable is more than a shortcut. It starts a bundled Python backend sidecar, generates a desktop-only token, chooses an available localhost port, and opens the React app served by that sidecar. Runtime files live under the platform app-data directory, not inside the repository checkout.

During tauri dev, the desktop shell uses the fixed backend port 8000 because Vite serves the React app and proxies API calls to a known target. Packaged builds use a dynamic backend port and navigate the window directly to the sidecar-served app.

Migration Sequence

  1. Keep the current backend working while adding the React app at /app-next.
  2. Move dashboard, models, providers, traffic, settings, backups, logs, and chat into React one surface at a time.
  3. Replace legacy Python HTML pages once the React app reaches feature parity.
  4. Package the backend as a sidecar and let Tauri control process lifecycle and app data paths.
  5. Move runtime state out of the source tree and into platform app data.

Current Milestone

The desktop shell now uses Tauri with a bundled Python sidecar and serves the same proven desktop control plane at /app-next that the legacy pywebview app used at /app. The React workspace remains available for future UI work, but the packaged desktop experience intentionally keeps feature parity with the original app.
